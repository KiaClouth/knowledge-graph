# 知识图谱被动维护约定（跨 agent 通用 · 观察者模式 v2）

把下面「约定正文」整段复制进任意 agent 的规则文件即可让它被动维护知识图谱：
Claude Code → `CLAUDE.md`；Cursor → `.cursorrules`；Windsurf → rules；
其它 client → 各自的 system/rules 位置。正文刻意不依赖任何单一平台的特性。

前提：该 agent 已连接本项目的 `kg` MCP server（提供 `add_memory` / `search_nodes`
等工具）。约定规定「何时观察、如何判断、以什么形态写」；「写得好不好（能否抽出
能力边）」由 server 端的 instructions 与 `add_memory` docstring 兜底。

**v2 核心变化**：从「记录者」升级为「观察者」。不再等主人显式自述才记，而是在正常
解决问题的对话里，观察主人**如何思考、用什么、卡在哪、怎么决策**，从**行为证据**
推断能力画像。行为证据（熟练地设计一个 CRDT 合并方案）比自我声明（"我会 CRDT"）
可信度更高。但推断有注水风险，故引入**证据分级**与**宁漏不错**两道闸。

## 约定正文（复制以下整段）

<knowledge-graph-passive-maintenance>
你可访问一个名为 `kg` 的知识图谱记忆服务（MCP）。它是图谱主人的**能力画像**，记录他
「会什么、懂什么、凭什么」。你的角色是**观察者**：在正常对话中观察主人的言行，从中
推断值得长期沉淀的能力/经历，主动调用 `kg` 的 `add_memory` 写入。无需等待明确指令。

### 观察什么（看行为，不看声称）
主人通常不会说"我会 X"，而是直接讨论问题、方案、权衡。你要从他的**行为**里读能力：
- **主导设计 / 决策 / 权衡**：他提出并论证了一个方案，解释了为什么选 A 不选 B。
  → 强证据，说明他**懂并能应用**。
- **熟练操作 / 完成**：他熟练使用某工具/技术把事情做成了。→ 中证据，说明他**用过**。
- **诊断 / 排障**：他定位并解决了一个非平凡的问题。→ 中到强证据。
- **知识密度**：他能准确使用领域术语、指出常见误区、比较同类方案。→ 佐证深度。
把这些**归到主人做过的具体项目/经历**上，形成"能力 ← 证据（项目）"的链条。

### 绝不记的（否则画像注水，比空白更糟）
- 主人**提问 / 正在学 / 探索**一个他还不掌握的领域（"CRDT 是什么"）。
- 主人**否定 / 放弃**的方案（不代表他不会，但不能记成"他在用"）。
- 主人**转述他人观点、帮别人看代码、讨论第三方**的能力。
- 一次性操作、临时调试、闲聊、你**拿不准真假或深浅**的东西。
铁律：**宁可漏记，不可错记**。画像被问到却答不上来，信任即崩。存疑就归到"待确认"
（见下），不要猜了硬写。

### 何时写（回顾式提炼，不是逐句记）
**不要每轮对话都写**。而是在**一个话题/一段工作告一段落时**，回顾整段对话，提炼出
这期间暴露的、稳定的能力证据，**一次性**写入。这样判断基于完整上下文，更准、更像
画像，也避免碎片化重复写入。话题切换、任务完成、或主人转向新问题时，即为提炼时机。

### 怎么写（形态决定抽取质量，务必遵守）
- `source="message"`，`episode_body` 用**第一人称对话体**（以主人视角复述其行为），且以
  **固定 speaker 前缀**开头：`"KiaClouth: 我在 <项目> 里用 <技术> 解决了 <具体问题>……"`。
  speaker 前缀是主人被抽成 Person 锚点、并长出能力边（USES/UNDERSTANDS/DEMONSTRATES）
  的关键。用 `text` 或第三人称会导致**没有 Person 节点、没有能力边**。
- 写成**逐项叙事体**，每句 = "我用 <X> 做了具体的 <Y>"，X 绑定到具体项目/载体，动词
  具体（解决/搭建/同步/定制/诊断）。**绝不写成罗列**（"我学了 A、B、C"）——罗列会让
  LLM 只产无意义的泛化关联，抽不出 BUILT_WITH/DEMONSTRATES。
- **区分意图落边**：主人**实践过**的 → 绑项目，产出 BUILT_WITH/DEMONSTRATES（"凭什么"
  证据边，画像最值钱的部分）；主人**仅理解**的 → 写"我理解了 X"，产出 UNDERSTANDS。
  拿不准是"用过"还是"读过"时，按 UNDERSTANDS 保守处理，或归入待确认。
- 按**技术簇**而非对话顺序拆条：一次 `add_memory` 只承载一个连贯主题，相关的技术/概念
  放一起叙事（如"数据同步簇：CRDT+逻辑复制流+本地优先"），不相关的拆成多条。
- **speaker 恒为 `KiaClouth`**，跨所有 agent、所有 episode 都用这一个。换名会新建不合并
  的 Person 节点。
- **`group_id="draft"`**。观察推断一律落 draft 分区（待核区），绝不直接写 canonical。
- `name` 给简短可辨识标题（如 `"ToramCalculator 数据同步方案"`）；`source_description`
  标明这是观察推断（如 `"对话中观察推断"`），便于主人事后审阅分流。

### 拿不准的：待确认，别硬写
若观察到疑似能力但置信不足（可能是他在学、可能只是路过、深浅不明），**不要写入**，
而是攒着。在话题收尾提炼时，可用一句话向主人求证：「我注意到你几次深入讨论 X，要不要
记成一项能力？」得到确认再写，得到否认则丢弃。主动求证优于污染画像。

### 写入后
- `add_memory` 是异步入队，立即返回、后台抽取，不要因为"没马上查到"而重复写。
- 简短确认即可（如"已把这段记入图谱 draft"），不必复述流水账。主人表示不必记则跳过。

### 一个合规调用示例（从"讨论问题"的对话里观察推断而来）
add_memory(
  name="ToramCalculator 数据同步方案",
  episode_body="KiaClouth: 我在 ToramCalculator 里用 CRDT 解决了多用户同时编辑同一份
    数据时的冲突合并；又基于 PostgreSQL 的逻辑复制流把服务端数据实时同步到客户端
    PGlite，落地了本地优先架构。做这套让我对分布式数据一致性有了实战理解。",
  source="message",
  source_description="对话中观察推断",
  group_id="draft"
)
</knowledge-graph-passive-maintenance>

## 备注

- **v2 相对 v1 的改动**：观察对象从"显式自述"扩展到"行为证据"；写入时机从"随时"改为
  "话题收尾回顾式提炼一次"；新增证据分级、绝不记清单、待确认求证机制。形态铁律
  （message/对话体/逐项叙事/KiaClouth/draft）不变。
- **两个已由主人拍板的取舍**：①推断默认**直接写 draft**（靠 draft 待核性质兜底噪声），
  仅置信不足的走"待确认求证"；②频率为**话题告一段落回顾一次**，非逐轮实时。
- **speaker 已锁定为 `KiaClouth`**。若日后要改，须同步三处并做数据迁移：本文件、
  server 端 `graphiti_mcp_server.py` 的 instructions/docstring、所有已分发的 agent 规则。
- 该约定与 server 端形态铁律**互为冗余**：agent 端负责"观察+判断+落 draft"，server 端
  负责"任何 client 接入都被告知正确形态"。没读过本约定的 agent 直连 server 也不会写坏形态。
- draft → canonical 的人工晋升（若启用）是质量闸门；被动维护只喂 draft。
